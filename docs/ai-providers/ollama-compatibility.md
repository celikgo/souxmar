# Ollama compatibility matrix

Generated by `souxmar-eval-llm` against locally-installed Ollama
models. Re-run after pulling a new model:

```sh
souxmar-eval-llm evals/v1-llm \
  --provider ollama \
  --model    llama3.1:8b \
  --output   docs/ai-providers/ollama-llama3.1-8b.json
```

The compatibility matrix below is curated from the per-model JSON
output files in this directory. Each cell records the pass rate
(N passes / M tasks) plus a one-line judgement.

## Tested model families

| Family | Smallest viable | Recommended | Notes |
| ------ | --------------- | ----------- | ----- |
| Llama-3.x   | `llama3.1:8b`     | `llama3.1:70b` (if you have ≥ 40 GB VRAM) | Native tool-calling support from 3.1+; 3.0 lacks the function-calling head. |
| Qwen-2.x    | `qwen2.5:7b`      | `qwen2.5:32b`                              | Strong JSON adherence; tool-call argument shapes are reliably valid. |
| Mistral-Nemo | `mistral-nemo:12b` | `mistral-nemo:12b`                       | Single tested size; tool-call quality strong on first turn, drifts on long histories. |

Quantisation: Ollama defaults to `Q4_K_M` for 4-bit. Tested
configurations below use the default quant unless noted; the
`Q5_K_M` and `Q8_0` runs are documented separately in their
per-model JSON files.

## Task results (Sprint 10 push 9 snapshot — 2026-05-11)

These are the initial verification runs. Each pass rate is the
fraction of the v1-llm eval set the model passed across 3 independent
runs (temperature 0; the variance comes from KV-cache reset
non-determinism in Ollama < 0.3.10).

| Task | llama3.1:8b | llama3.1:70b | qwen2.5:7b | qwen2.5:32b | mistral-nemo:12b |
| ---- | ----------- | ------------ | ---------- | ----------- | ---------------- |
| llm-01-list-plugins  | 3/3 | 3/3 | 3/3 | 3/3 | 3/3 |
| llm-02-mesh-no-bcs   | 2/3 | 3/3 | 3/3 | 3/3 | 2/3 |

The flaky `llm-02-mesh-no-bcs` runs on `llama3.1:8b` and `mistral-nemo`
both hallucinated a `solve` tool call once across the three runs. The
70B Llama and both Qwen sizes never tripped this — a result that
matches the general "function-calling correctness scales with parameter
count" observation across the literature.

## Known unsupported models

* **Llama-3.0** — no native tool-calling head; the model emits
  free-form text describing tool calls instead of structured ones.
  Skipped by `souxmar-eval-llm`.
* **gemma2** — Ollama's function-calling support for Gemma 2 is
  experimental as of Ollama 0.3.x; pass rates oscillate widely
  between runs. Re-test after Ollama upstream stabilises.
* **Models < 7B parameters** — too small to reliably emit
  schema-conforming tool-call JSON. Pass rates below 50% across
  every task tested; not recommended for production use.

## Hardware requirements

| Model           | VRAM (min) | VRAM (recommended) | First-token latency target |
| --------------- | ---------- | ------------------ | -------------------------- |
| llama3.1:8b     | 8 GB       | 12 GB              | ≤ 500 ms                   |
| llama3.1:70b    | 40 GB      | 48 GB              | ≤ 1500 ms                  |
| qwen2.5:7b      | 8 GB       | 12 GB              | ≤ 500 ms                   |
| qwen2.5:32b     | 24 GB      | 32 GB              | ≤ 1000 ms                  |
| mistral-nemo:12b | 16 GB     | 20 GB              | ≤ 800 ms                   |

These are the lower bounds; faster hardware obviously runs the
models faster. The first-token latency targets line up with
`docs/ENGINEERING_PRACTICES.md`'s "first chat token < 800 ms p95"
budget for the BYOK / local path.

## Re-running the matrix

`souxmar-eval-llm` accepts `--output <path>` to emit a JSON summary
per (provider, model) run. The summary is the source of truth this
markdown table is curated from. A future push (Sprint 11+) wires a
`souxmar-eval-llm --matrix` mode that runs every installed model
against the full task set and regenerates the table directly.

For automation today:

```sh
for model in llama3.1:8b qwen2.5:7b mistral-nemo:12b; do
  souxmar-eval-llm evals/v1-llm \
    --provider ollama \
    --model    "${model}" \
    --output   "docs/ai-providers/ollama-${model//[:.]/-}.json"
done
```

Then update this matrix from the per-model JSON files.
