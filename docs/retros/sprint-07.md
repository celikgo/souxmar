## What to keep

- **The soak-then-freeze ritual worked end-to-end.** ADR-0007 declared the candidate; the conformance suite stayed green across Sprint 6; one additive ratchet event landed cleanly in Sprint 6 push 4 and another in Sprint 7 push 3; ADR-0008 closed the freeze. The CI gate (`scripts/check-frozen-headers.sh`) accepted both ratchet events with the expected commit-message marker. **The mechanism is the contract.**
- **The "always-on + opt-in" plugin pattern is paying off.** Every new external dependency (OCCT in Sprint 6 push 4, Gmsh in Sprint 6 push 5, DOLFINx in this sprint's push 2) ships behind a `SOUXMAR_WITH_*` build gate alongside a working closed-form / stub plugin that exercises the same capability namespace. Default CI keeps a fast, deterministic surface; nightly CI exercises the heavy adapters. This generalises to OpenFOAM next sprint.
- **The agent eval suite caught its target failure mode immediately.** The 30 v1 tasks include both success paths and structured-error paths — `NOT_AVAILABLE` / `INVALID_ARGUMENT` / `PLUGIN_NOT_FOUND` / `PRECONDITION_FAILED`. Pinning the dispatcher's structured-error surface as a CI-enforced contract is exactly the discipline the LLM-driven Sprint 8+ work needs to inherit.

## What to fix

- **The `SOUXMAR_E_PARSE` typo from Sprint 6 push 4.** When writing the STL reader I assumed an error code that didn't exist; the fix was `s/SOUXMAR_E_PARSE/SOUXMAR_E_IO/g`. The status.h enumeration is too small — `IO` covers both "filesystem error" and "file-content malformed," which conflates two operationally-distinct error classes. **Action: open an additive-minor ratchet PR for Sprint 8 adding `SOUXMAR_E_FILE_FORMAT` alongside the existing codes.** Touches `status.h` (frozen) — uses the additive marker.
- **macOS bash 3.2 surfaced late in the freeze gate.** The Sprint 7 push 1 `scripts/check-frozen-headers.sh` initially used `mapfile` (bash 4+) and broke on a macOS local test. Fixed inline (push 3 commit `991731b`). **Action: add a `scripts/_compat.md` documenting "every script must run under bash 3.2."**
- **The FEniCSx adapter ships a `poisson.c` that isn't committed.** It's machine-generated by FFCx; the build STATUS-skips when it's absent. Long-term we want either (a) FFCx at build-time as a hard dep when `SOUXMAR_WITH_FENICSX=ON`, or (b) the file committed under a clearly-marked generated-artefact directory. **Action: surface this in the Sprint 8 retro after we see how the OpenFOAM adapter handles its analogous "case files are generated" problem.**

## One ADR-worthy decision surfaced

**Subprocess plugins as a first-class category.** The OpenFOAM ADR-0009 work made it concrete: subprocess-isolated plugins are sufficiently different from in-process plugins that the plugin host's surface should distinguish them. A subprocess plugin doesn't need `guard_call` (the OS process boundary is the guard), doesn't fit cleanly into the `single-threaded` / `internal-parallel` / `reentrant` threading model (the threading lives in another process), and has different observability requirements (stderr/stdout are the audit log inputs, not the host's internal logger).

We don't need to act on this today — the OpenFOAM plugin can be the proof-of-concept for the pattern, and ADR-0010 (Sprint 8 or 9) can formalise it as a `plugin.execution` manifest field once we've shipped the first real example.
