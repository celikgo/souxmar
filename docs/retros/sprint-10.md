# Sprint 10 retro — Distribution + plugin marketplace v0 (auto-updater stack end-to-end + Ollama provider verified + desktop onboarding scaffolded + fourth example)

**Closed:** 2026-05-11. **Pushes:** 12. **Theme:** "make the project shippable, make the ecosystem joinable."

Sprint 10 was the distribution sprint. Eight of the eleven SPRINT_PLAN.md sprints to internal alpha are done; the remaining two (S11 dogfood, S12 public alpha) ride on the surfaces this sprint locked in. The headline outcome: souxmar is no longer "a CLI + plugin library that compiles on three OSes" — it's "a CLI + plugin library + onboarding-wizard-scaffolded desktop app + signed auto-update channel + publishable plugin marketplace, ready for internal alpha." The auto-updater end-to-end (manifest → verifier → state machine → apply → rollback → release signing) is the load-bearing thing; everything else this sprint orbited around making that arrive on a user's machine intact.

## What to keep

- **The "ADR locks the design before the code" pattern, applied at XL-story scale.** ADR-0013 (push 4) locked in the manifest format + the signing scheme + the alternatives-considered + the pre-mortem before a single byte of crypto code landed. Pushes 5-8 then implemented straight against that ADR — every reviewer comment on those pushes was about behaviour, not data-model litigation. The ADR-as-design-contract approach scales further than expected: a 4-push XL story is still tractable when the contract document does the up-front work. Reproducible for future XL stories (managed-AI proxy Sprint 14; cloud sync Sprint 15; marketplace v1 Sprint 16).
- **Splitting parse from verify from gate, deliberately.** Manifest parser tests run without crypto (push 4 → 24 tests, no libsodium). Verifier tests run without filesystem (push 5 → 27 tests, no on-disk fixtures). State-machine tests run without wall-clock (push 6 → ~30 tests, every one against a `FixedTimeSource`). The three test surfaces each isolate one concern; a refactor that breaks one doesn't ripple. Mirrors the "compose-from-units" architectural style ADR-0001 articulated for plugins; the auto-updater stack is the same idea applied to a single feature.
- **The signature-as-typed-enum approach.** Every place we'd be tempted to return a string ("signature bad: wrong key length"), we return an enum value. `SignatureStatus`, `RefusalReason`, `ApplyOutcome`, `RollbackOutcome`, `ProviderErrorKind` — five enums across three modules, each surfaced as an integer in the on-disk rollback log + each pinned by a `to_string()` test. A refactor that drops or renames a value is a test failure, not a downstream log-parsing breakage. The audit-log story this enables — "exactly why did this update refuse on 2026-08-12?" answerable by a single byte — is load-bearing for the Sprint 11+ support flow.
- **The marker-file-not-symlink choice.** Push 7's atomic-switch via renamed marker files (`current.txt`) instead of a symlink dance keeps the cross-OS code path single-file-write-rename on all three OSes. No Windows junction-point dance, no symlink-permission tangling, no readlink fork. The OS-rename-is-atomic invariant is more uniformly true than the symlink-rename-is-atomic invariant; preferring the boring path paid off.
- **Subprocess-based HTTP at the AI provider layer.** Push 9's OllamaProvider shells out to `curl` rather than linking an in-process HTTP client. The Sprint 8 push 1 subprocess harness was already in the tree for OpenFOAM; reusing it kept libsouxmar-ai dependency-light at the cost of a process spawn per inference. The spawn cost (~5 ms on Linux, ~15 ms on macOS) is negligible against typical LLM inference latency (seconds). Sprint 11+ can revisit if the managed-AI proxy needs sub-100ms calls.

## What to fix

- **Desktop onboarding wizard isn't visually verified.** Push 10 lands the React components + token-driven styling + Tauri commands; the test harness that builds the app, screenshots the wizard, and compares against a baseline is queued for Sprint 11. Per push 10's commit message this is *honestly* a follow-up — the alternative would be landing a screenshot baseline before the wizard has been seen by a human, which would gate against the wizard itself. **Action: Sprint 11 push 1 builds the visual-regression harness (Playwright + Tauri build) + commits the first set of dim-theme baselines.** The dogfood week (Sprint 11 theme) will surface the rough edges either way; the question is whether the visual gate catches a future regression cleanly.
- **HTTPS fetcher for the auto-updater isn't wired.** Push 7's `apply` takes `--artifact <local-path>`; the desktop app (or a future `souxmar update fetch`) is expected to download separately. The signature path's trust boundary is honest — bytes get hashed against `manifest.artifact.sha256` regardless of where they came from — so this isn't *unsafe*, it's *incomplete*. **Action: Sprint 11 push N adds `souxmar update fetch` that talks HTTPS via the subprocess-curl approach OllamaProvider already uses (push 9 validated that pattern).** The fetched bytes flow into the same `apply_update` call as today's `--artifact` path; the trust path is unchanged.
- **The embedded release key is a placeholder.** Push 8's `SOUXMAR_RELEASE_PUBKEY_HEX` cache var defaults to a 64-char placeholder (not a real ed25519 public key — arbitrary bytes). The dev-key regeneration script (`scripts/release/regenerate-dev-key.sh`) writes a real keypair to a gitignored file when run; the release CI workflow injects the real release-2026 key. So no security gap exists *in production*. But the placeholder being a placeholder means a developer running `souxmar update apply --no-trusted-key` against a hand-signed manifest will see a verifier mismatch with no helpful "you forgot to regenerate the dev key" hint. **Action: regenerate-dev-key.sh runs automatically as part of `cmake --build` when `SOUXMAR_RELEASE_PUBKEY_ID=souxmar-dev-key` AND `dev-signing-key.local.txt` doesn't exist; the CMake side writes the matching pubkey into the cache directly.** ~3 pts; queued for Sprint 11.
- **The mesh-comparison study's report rendering is unverified at end-to-end scale.** `examples/mesh-comparison/compare.py` parses per-cell quality DataArrays via regex against the VTU XML — it works on the in-tree writer.vtu output, but a third-party `writer.vtu`-conforming plugin might emit subtly different XML (whitespace, attribute ordering). **Action: Sprint 12+ adds a small "VTU consumer conformance" check the writer adapters run against — locks the DataArray emission shape that compare.py + ParaView + the plugin-marketplace mesher-evidence renderer all rely on.** Not Sprint 11 — Sprint 11 is dogfood; this is a longer-tail thing.
- **Sprint 10 ran very hot.** 12 pushes against a 65 pts forecast. The auto-updater XL story alone consumed pushes 4-8 (~7,500 LOC measured) — that's roughly the size of Sprint 6 in its entirety. The capacity model has now seen two consecutive sprints (9, 10) substantially above forecast; Sprint 11 should plan against ~50 pts conservatively (split the difference between the 75-80 measured at S9 and the dogfood-week light-touch expectation). **Action: Sprint 11 forecast is 50 pts; explicit acknowledgement in the next retro if velocity stays high so the trend doesn't go un-recalibrated indefinitely.**

## One ADR-worthy decision surfaced

**The "signed manifest + embedded trust store + yearly rotation" pattern is now a project-wide trust paradigm, not an auto-updater-only one.** ADR-0013 designed it for the update channel; ADR-0014 ratified the rotation cadence. Two sprint themes from S13+ that *should* reach for the same shape:

- **Plugin marketplace verification (Sprint 16).** Right now a plugin's authenticity is "the conformance badge is in the index TOML." That works for an open repository but doesn't work for paid plugins (where a malicious mirror could substitute the .so). The natural fix: every marketplace plugin's manifest carries an ed25519 signature against a per-author key; the marketplace's *meta-manifest* (the index file itself) is signed by the souxmar release key. Reuses the verifier from push 5, the trust-store from push 8, the embedded-key-injection mechanism from push 8.

- **Cloud sync E2E encryption (Sprint 15).** Pro-tier cloud sync needs an out-of-band trust anchor for the per-project encryption keys. The yearly-rotation + embedded-pubkey + key-id pattern is exactly the shape that fits — replace "release public key" with "project-owner-Ed25519-public-key" and the same code path verifies.

The decision worth ADR'ing isn't the cryptographic mechanism (that's covered) — it's whether to **factor the verifier + trust store into a `libsouxmar-crypto` shared module** before the second and third users land. Doing it now (Sprint 11 push 1?) before plugin marketplace + cloud sync depend on the auto-updater's surface is cheaper than refactoring later. Queueing the ADR as "0015-crypto-module-extraction" for the Sprint 11 retro round.

## Outcome

souxmar is at **v0.9.0-beta4** as of this commit. The fourth public pre-release closes the auto-updater XL story across pushes 4-8 (manifest → verifier → state machine → apply → rollback → release signing), opens the plugin marketplace v0 surface (data model + search + publication workflow), verifies the local-Ollama AI provider across three model families, scaffolds the desktop onboarding wizard, and ships the fourth in-tree example. The C ABI stays at v1.3 frozen final (ADR-0008 + ADR-0012); the tool contract stays at v1 frozen final with 18 tools (ADR-0011). The release-signing trust path is wired end-to-end — embedded pubkey + manifest signature + per-platform OS signing (macOS notarisation, Windows EV, Linux GPG) — and documented in docs/SECURITY.md + ADR-0014.

The desktop app is still not user-facing — the workbench shell beyond the onboarding wizard is empty. The Sprint 11 dogfood week is the next exit criterion: every engineer runs a real analysis using only the desktop app + chat. Sprint 12 then cuts the public alpha. None of those undermine the contract this sprint shipped: **the auto-updater (and the cryptographic trust paradigm underneath it) is operationally complete; the plugin marketplace data model + publication workflow exist; the AI provider abstraction is in place for managed-AI in S14; and Sprint 11 has a concrete dogfood target rather than an aspirational one.**
